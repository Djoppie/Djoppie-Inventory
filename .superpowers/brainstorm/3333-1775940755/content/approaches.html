Aanpak: Cleanup & Herorganisatie

Twee strategieen voor het uitvoeren van de volledige herorganisatie

A

Fase-gebaseerd per Feature (Aanbevolen)

Werk feature per feature door het hele project heen. Elke fase pakt een feature-groep aan: verwijder dead code, splits monolithen, herorganiseer backend + frontend + routes + docs.

Fase 0: Dead code cleanup (ongebruikte files/componenten/docs verwijderen)
Fase 1: Inventory (assets, templates, QR, CSV import/export)
Fase 2: Workplaces (physical workplaces, split monolith)
Fase 3: Operations (rollouts, requests, swaps)
Fase 4: Devices (Intune split, licenses, software)
Fase 5: Reports (split monolith, per-feature reports)
Fase 6: Admin (organization, locations, types, RBAC prep)
Fase 7: Dashboard + User + cross-cutting concerns
Fase 8: Docs, CLAUDE.md update, final cleanup

Voordelen

  • Elke fase is testbaar en deploybaar
  • Feature-context blijft behouden tijdens werk
  • Regressie beperkt tot een feature-domein

Nadelen

  • Duurt langer voordat alles klaar is
  • Tijdelijk mix van oude en nieuwe structuur
B

Laag-gebaseerd (Horizontaal)

Werk per laag: eerst alle dead code weg, dan alle backend controllers herorganiseren, dan alle frontend pages, dan alle routes, dan docs.

Fase 0: Verwijder alle dead code, ongebruikte files, archived docs
Fase 1: Backend controllers hergroeperen + monolithen splitsen
Fase 2: Backend services/repos hergroeperen
Fase 3: API routes hernoemen
Fase 4: Frontend pages/components hergroeperen
Fase 5: Frontend routing + navigatie
Fase 6: Docs + CLAUDE.md

Voordelen

  • Snellere initiële cleanup
  • Consistentie per laag gegarandeerd

Nadelen

  • Groter regressierisico (veel tegelijk kapot)
  • Moeilijker te testen tussen fasen
  • Frontend en backend tijdelijk uit sync

Mijn aanbeveling: Optie A

Feature-gebaseerd werken houdt het risico beperkt en elke fase levert een werkend resultaat op. Fase 0 (dead code cleanup) is een snelle win die direct overzicht geeft. Daarna kan elke feature-fase onafhankelijk getest en gedeployed worden.
