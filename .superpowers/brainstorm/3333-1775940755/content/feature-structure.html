Voorgestelde Feature-Structuur

Hoe de navigatie en code georganiseerd worden rondom de 7 hoofd-features

A

Volledige Herorganisatie (Aanbevolen)

Routing, mappenstructuur en navigatie allemaal herorganiseren naar feature-groepen. Eenmalig meer werk, maar resulteert in een consistente, schaalbare structuur.

/dashboard — KPI's, overzicht
/inventory/assets, /templates, /import
/workplaces/overview, /:id
/operations/rollouts, /requests, /swaps
/devices/overview, /intune, /licenses, /software
/reports/hardware, /workplaces, /operations
/admin/organization, /locations, /asset-types
B

Alleen Code-Structuur

Mappenstructuur in de code herorganiseren (pages/inventory/, pages/operations/, etc.) maar de huidige URL-routes behouden. Minder risico, maar mismatch tussen URL en code.

URLs blijven: /assets, /rollouts, /workplaces, /templates, etc.
Code wordt: pages/inventory/AssetsPage.tsx, pages/operations/RolloutPage.tsx
C

Minimaal - Alleen Cleanup

Huidige structuur behouden, alleen dead code verwijderen en monolithen splitsen. Snelst, maar lost de organisatie-problemen niet op.
